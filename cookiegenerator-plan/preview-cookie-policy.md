## Context

The cookie-policy generator (`/cookie-policy/:websiteId`) lets the user edit three
sections (About cookies, Use of cookies, Cookie preferences) plus an effective
date, but there is no way to see the composed policy as it would appear on a real
page. Assignment requirement **2.4 Preview** says: "At any point (and required
before final generation), the user can preview how the policy will look, rendered
as it would appear on a real page — not just a form dump." CookieYes does this
with a "Preview cookie policy" button pinned at the bottom of the sidebar that
opens a "Policy preview" modal.

## Objective / feature request

Add a **Preview cookie policy** button at the bottom of the Cookie Policy sidebar.
Clicking it opens a **Policy preview** modal rendering the full policy (title,
effective/last-updated dates, each section's heading + rich-text HTML including
uploaded images, the Consent Preferences button and browser links that live inside
the section content, and a "generated by" footer) — a real-page rendering, not a
form dump.

## Specifications

- Sidebar (`<aside className="cp-side">`) gets a "Preview cookie policy" button
  pinned at the bottom, below the section nav, with a document icon (inline SVG),
  matching the CookieYes reference screenshot. Always enabled.
- Clicking it opens a full-overlay modal:
  - Header row: bold title **Policy preview** + close (✕) button; divider below.
  - Scrollable body rendering, in order:
    1. `Cookie Policy` (h1)
    2. `Effective date: <formatLong(effectiveDate)>` (falls back to today if unset)
    3. `Last updated: <formatLong(todayISO())>`
    4. For each of the three sections in order (about → use → preferences):
       section `heading` (h2) + its `description` HTML rendered via
       `dangerouslySetInnerHTML`. Sections whose heading and description are both
       empty are skipped.
    5. Footer line: `Cookie Policy generated by <site url>` (muted).
- The preview renders the **current in-editor state** (`data`, `effectiveDate`),
  including unsaved edits — satisfies "at any point".
- Uploaded images inside descriptions (`<img src="/pulse/images/<uuid>">`) render
  as-is (public GET endpoint — no auth work needed).
- Close via: ✕ button, clicking the overlay backdrop, or Esc key. Body scroll is
  locked while open.
- No new API endpoint, no backend/openapi change.

## Requirement alignment

- **2.4 Preview** (from the assignment's policy-generator section, per the user's
  screenshot): fully satisfied — rendered-page preview available at any point.
  The "required before final generation" clause is out of scope here because no
  "final generation" step exists yet; noted as a follow-up gap for whichever
  feature adds final generation (it should gate on having opened the preview).
- R1–R8 (consent/CMP requirements in `references/assignment.md`): not touched.
  Core principle not violated — no cookies created; this is pure rendering of
  stored content.
- Gap flagged: the CookieYes reference shows 5 sidebar items (Language settings,
  Types of cookies, …) and cookie tables; our app has only 3 sections and no
  cookie-table data model, so the preview renders exactly the 3 real sections.

## Design

All in `frontend/` (React 19 + Vite, plain JS, plain CSS in `src/signup.css`):

1. **`frontend/src/PolicyPreview.jsx`** (new, ~60 lines) — props
   `{ url, data, effectiveDate, onClose }`. Renders overlay
   `<div className="cp-modal-overlay">` + `<div className="cp-modal" role="dialog"
   aria-modal="true" aria-label="Policy preview">`. Uses `formatLong` / `todayISO`
   from `src/dateUtils.js`. `useEffect` for Esc listener + `document.body`
   scroll-lock. Rendered inline at the end of `CookiePolicyPage`'s tree (same
   pattern as the toast at `CookiePolicyPage.jsx:481`) — no portal, consistent
   with existing code.
2. **`frontend/src/CookiePolicyPage.jsx`** — add `const [showPreview,
   setShowPreview] = useState(false)`; add the button after `</nav>` inside the
   `<aside>` (around line 320) with class `cp-preview-btn`; render
   `{showPreview && <PolicyPreview … onClose={() => setShowPreview(false)} />}`.
   Section order/labels come from the existing `SECTIONS` array (lines 14–44) so
   the preview stays in sync with the editor.
3. **`frontend/src/signup.css`** — new `cp-` prefixed block: `.cp-preview-btn`
   (soft accent pill pinned via `margin-top:auto` in the flex sidebar),
   `.cp-modal-overlay` (fixed, dimmed), `.cp-modal` (white card, large,
   scrollable body), `.cp-modal-head`, `.cp-preview-body` typography that mirrors
   the existing `.rte-content .ProseMirror` rules (`signup.css:1053–1072`) so
   paragraphs, lists, links, and `img { max-width:100% }` look identical to the
   editor's rendering. Reuse existing tokens (`--accent`, `--border`, `--muted`).
4. **Docs:** add `frontend/AI_DOCS/policy_preview.md`; run `sync-claude-md` after
   implementation to update `frontend/CLAUDE.md` (Structure tree only — no route
   or endpoint changes).

## Design notes

- **First `dangerouslySetInnerHTML` use in the codebase.** Acceptable here: the
  HTML is the user's own Tiptap output, already stored/rendered by the editor
  itself; same trust boundary as the editor.
- **Last updated = today** (`todayISO()`), matching CookieYes behaviour in the
  reference screenshot (effective July 08, last updated July 09 = view date). The
  backend doesn't expose the row's `updated_at`; adding it was rejected as
  unnecessary backend surface for a preview.
- **Modal, not new tab** — matches the reference screenshots; keeps unsaved state
  available without persisting.
- **No portal** — nothing in the app uses `createPortal`; inline overlay at the
  page root has no stacking-context conflicts in this layout.
- The "Consent Preferences" button and Chrome/Safari/Firefox/IE help links appear
  in the preview automatically because they are part of the seeded
  `cookiePreferences` section HTML (`backend/src/utils/defaultCookiePolicy.js`)
  — no dedicated components needed.

## Acceptance criteria

- [ ] "Preview cookie policy" button visible at the bottom of the sidebar on
      `/cookie-policy/:websiteId`, styled like the reference (icon + soft pill).
- [ ] Clicking it opens the "Policy preview" modal; ✕, backdrop click, and Esc
      all close it; page behind doesn't scroll while open.
- [ ] Modal shows: "Cookie Policy" title, `Effective date: <long date>`,
      `Last updated: <today, long>`, then each non-empty section's heading and
      formatted rich-text content (bold/lists/links/images render, not raw HTML
      — i.e. a real page rendering, not a form dump).
- [ ] An image uploaded via the editor appears rendered inside the preview.
- [ ] Unsaved edits (typed but not saved) appear in the preview.
- [ ] "Consent Preferences" button + browser help links visible in the Cookie
      preferences part of the preview (from seeded content).
- [ ] Frontend build + lint pass; no backend changes (`git -C backend status`
      clean apart from the plan doc).
